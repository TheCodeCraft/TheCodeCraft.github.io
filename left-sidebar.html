Asynchronous Execution in JS Explained

As a developer, it's crucial to grasp the nuances of asynchronous programming in JavaScript. One common source of confusion is the order of execution when dealing with asynchronous operations. In this article, we'll explore a scenario that often traps developers and dive into the inner workings of the event loop.

The Problem: Asynchronous Order Mysteries

Consider the following code:

console.log("Start");
setTimeout(() => {
  console.log("Timeout callback");
}, 0);

fetch('https://jsonplaceholder.typicode.com/todos/1')
  .then(response => response.json())
  .then(data => {
    console.log("Fetch success:", data);
  });
console.log("End");

Challenge

Before running the code, take a moment to think about the expected order of the console log statements. What might surprise you about the asynchronous nature of JavaScript?

The Expected Output:

Start
End
Fetch success: { /* data from the API */ }
Timeout callback

Let's break down the execution sequence:
console.log("Start"): The initial log statement is executed synchronously.
fetch Operation: Initiates an asynchronous HTTP request, but the code doesn't wait for it to complete. Instead, it proceeds to the next line.
console.log("End"): Another synchronous log statement.
fetch Callback: fetch operation completes, and its then callback executes asynchronously when the data is available.
setTimeout Callback: The timeout callback, scheduled with a delay of 0 milliseconds, is executed last.
